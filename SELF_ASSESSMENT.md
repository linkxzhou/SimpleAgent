## 发现的问题（按优先级排序）

### 1. [中] pytest 警告信息影响用户体验

**现象：**
- 运行 `python -m pytest tests/` 时产生弃用警告
- `PytestDeprecationWarning: The configuration option "asyncio_default_fixture_loop_scope" is unset.`

**影响：**
- 开发者看到警告可能误以为代码有问题
- 降低专业感

**位置：** `pytest.ini` 或 `pyproject.toml`

**建议修复：**
```ini
# pytest.ini
[pytest]
asyncio_default_fixture_loop_scope = function
```

**修复成本：** 低（一行配置）

---

### 2. [中] 工具超时提示缺失

**影响：**

**建议修复：**

**归属：** ROADMAP.md 级别 2 已有记录

## 实际测试发现（第 10 小时）

### 直接体验问题

#### 1. [高] 缺少编辑预览 — 破坏性操作无确认

**现象：**
- `edit_file` 直接修改文件，不显示 diff 预览
- `write_file` 直接覆盖文件，无确认提示
- 无法在应用更改前查看影响范围

**影响：**
- 用户可能在不确定的情况下破坏文件
- 无法在应用前检查 old_content 是否精确匹配
- 降低开发者信心

**归属：** ROADMAP.md 级别 2 "差异预览" 任务

---

#### 2. [高] 缺少撤销能力 — 无法恢复误操作

**现象：**
- 编辑或覆盖后无法撤销
- 没有 `/undo` 命令
- 没有自动备份机制

**影响：**
- 误操作可能导致数据丢失
- 降低开发者信任度
- 无法安全试验

**归属：** ROADMAP.md 级别 2 "/undo 命令" 任务

---

#### 3. [中] 缺少操作确认机制

**现象：**
- 覆盖性操作（write_file、edit_file）无确认步骤
- 不能设置"确认模式"
- 无法区分安全操作和危险操作

**影响：**
- 不小心运行危险脚本可能破坏重要文件
- 与真实开发流程不符（通常有 git diff 等确认步骤）

**建议：**
- 添加 `--confirm` 模式，破坏性操作前询问
- 或自动为 .py 文件等关键文件类型添加确认

---

### 流程摩擦点

#### 4. [低] 清理步骤需要手动执行

**现象：**
- 测试文件创建后，需要手动执行 `rm` 命令清理
- 没有自动清理机制

**影响：**
- 低，符合开发者预期
- 但可能忘记清理，留下临时文件

**建议：**
- 可考虑添加临时文件命名约定（如 `.tmp_` 前缀）
- 或在 session 结束时提示未清理的文件
