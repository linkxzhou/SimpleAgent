# 代码简化专家

一个专注于提升代码清晰度、一致性和可维护性的代码简化技能，在保持所有功能完全不变的前提下优化代码表达。

## 使用场景

- 用户请求简化、精炼或优化代码
- 用户希望提升代码可读性和可维护性
- 用户要求去除冗余代码或不必要的复杂度
- 用户想要统一代码风格和命名规范
- 在完成功能开发后进行代码质量提升

## 核心原则

### 1. 保留功能完整性

严格保证代码行为不变。所有原有功能、输出和行为必须完整保留，只优化实现方式，不改变实现结果。

### 2. 遵循项目编码规范

按照项目已有的编码标准执行优化，包括但不限于：

- 模块导入排序与路径规范
- 函数声明风格（优先使用 `function` 关键字而非箭头函数）
- 顶层函数添加显式返回类型注解
- React 组件使用显式 Props 类型定义
- 错误处理模式（尽量避免 try/catch，优先使用显式错误处理）
- 命名规范保持一致

### 3. 提升代码清晰度

通过以下方式简化代码结构：

- 降低不必要的复杂度和嵌套层级
- 消除冗余代码和过度抽象
- 使用清晰的变量名和函数名提升可读性
- 合并相关逻辑
- 移除描述显而易见内容的多余注释
- **避免嵌套三元运算符** — 多条件场景优先使用 switch 语句或 if/else 链
- **选择清晰而非简短** — 显式代码往往优于过度紧凑的代码

### 4. 保持优化平衡

避免过度简化导致以下问题：

- 降低代码清晰度或可维护性
- 产生难以理解的"聪明"写法
- 将过多职责合并到单个函数或组件
- 移除有助于代码组织的合理抽象
- 为追求"更少行数"而牺牲可读性（如嵌套三元、密集单行）
- 使代码更难调试或扩展

### 5. 聚焦修改范围

默认只精炼当前会话中近期修改或触及的代码，除非用户明确要求审查更广范围。

## 执行流程

1. **识别目标代码** — 定位近期修改的代码段
2. **分析优化机会** — 评估代码在优雅性和一致性方面的提升空间
3. **应用项目规范** — 按照项目特定的最佳实践和编码标准执行优化
4. **验证功能不变** — 确保所有功能保持完整
5. **确认简化效果** — 验证精炼后的代码更简洁、更易维护
6. **记录重要变更** — 仅记录影响理解的关键改动

## 注意事项

- 优先处理可读性和可维护性，避免追求代码行数最少化
- 对于不确定是否应简化的部分，保持原样
- 大规模重构前应征得用户确认
- 保留有意义的注释和文档字符串
